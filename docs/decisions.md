# Decisions

> Locked technical decisions that explain *why the code is the way it is*. If you (human or AI) are about to "fix" something that looks wrong, check here first — it may be intentional. This is not a history of how we got here; it states what is true now.

## Container runtime

| Decision | Choice | Why |
|---|---|---|
| Container runtime | **Podman**, rootless | No daemon, rootless by default, Quadlet for systemd integration |
| Unit format | **Quadlet** `.container`/`.network`/`.volume` files | Systemd-native, human-readable, generated by the compiler |
| Orchestration | Quadlet in production; Podman Compose export for local dev/inspection | Never Kubernetes |
| Reverse proxy | **Caddy** | Automatic TLS, simple config |
| Caddy backend addressing | Routes address the backend by Podman container name (`<container-name>:<port>`), never `localhost:<port>` | Caddy runs isolated on the `ownbase-internal` network and cannot reach host-loopback ports; `localhost` silently produced dead routes |
| Caddy published ports | `HostPublishPorts: [80, 443]` only when at least one service has a domain configured (`schema.OwnbaseConfig.HasPublicDomain`) | A domain-less Base (the default state of a fresh Base) has nothing for Caddy to route yet; see "Local development" below for how such a Base is reached instead |
| Network/volume service management | `systemctl restart` (not `start`) for `.network`/`.volume` units | These are `Type=oneshot RemainAfterExit=yes` — `start` is a no-op once they've run once, even if the resource was later pruned. `restart` forces recreation. |
| `Environment=` quoting | Values with spaces are double-quoted by the compiler | Systemd splits unquoted values on spaces into separate `KEY=VALUE` pairs |
| Reload trigger | `SIGHUP` to PID 1, not `systemctl daemon-reload` | `daemon-reload` deadlocks when called from inside a systemd service transaction |
| Capabilities | `DropCapability=ALL` on every container; services opt back in via `add_capabilities:` in `ownbase.yaml` | Most restrictive default; exceptions are explicit and code-reviewed |
| Privilege model | Rootful: units in `/etc/containers/systemd`, system `systemctl`. Rootless: `~/.config/containers/systemd` + `systemctl --user`. Detected at runtime; both the core bootstrap and the per-service applier use the same split | The daemon runs as root for host hardening; user-level Quadlet needs a D-Bus session that doesn't exist for root outside a login context |

## Repository and Git

| Decision | Choice | Why |
|---|---|---|
| Authoritative repo | Bare git repo at `/opt/ownbase/repo` (config) + one bare repo per service under `/opt/ownbase/repos/` | User-owned, no third-party dependency, no git host to run or secure |
| Git UX layer | None — `ownbasectl` pushes directly to the bare repos over the normal admin SSH user, exactly like a manual `git push` | A hosted git UI (Forgejo) was tried and removed: it was a second attack surface, a second thing to back up and reconstruct, and added no capability `ownbasectl` + local bare repos didn't already provide — see the "Forgejo removal" entry below |
| Sync model | `post-receive` hook on the config bare repo sends `SIGUSR1` to the daemon, which pulls into the checkout and reconciles | Single signal path; no webhook, no second git host to keep in sync |
| Service source repos | `source:` services get an empty bare repo the user pushes into directly; `mirror:` services get a `git clone --bare --mirror` of the external URL, refreshed on demand when a new `ref:` is pinned (see `internal/repos`) | The Base is self-contained — the external git host is only consulted when a genuinely new ref appears, never on every reconcile |
| Repo ownership | Every bare repo is created by the daemon (root) and then chowned to the admin SSH user recorded at install time (`/opt/ownbase/admin-user`, see `internal/fsowner`) | The daemon runs as root, so a repo it creates is unwritable by any other account by default; chowning is the minimal fix that makes the documented "just `git push`" workflow actually work without a dedicated git-SSH proxy |
| Forgejo removal | Forgejo (hosted git UI + pull-mirror manager) was removed in favor of the above | It served three roles — config repo host, build source, mirror manager — all of which the existing bare-repo + hook mechanism already handled *underneath* Forgejo. Removing it cut a core package, a token, a webhook, and an entire AuthorizedKeysCommand SSH-proxy shim without losing any capability |

## Schema and compiler

| Decision | Choice | Why |
|---|---|---|
| Config format | YAML (`ownbase.yaml`) | Human-readable |
| Service declaration | `source:` (local bare repo path under `/opt/ownbase/repos/`) or `mirror:` (external git URL, auto-mirrored) — the only valid user service kinds | Every service builds from a local bare repo; no container registries as service sources |
| No-registry rule | `image:`/`digest:` do not exist on `ServiceDecl` | Enforced at the schema level, not just convention |
| Core packages | Caddy is installed by the installer, configured via the `core:` block, upgraded via `ownbasectl upgrade` — never declared as an `ownbase.yaml` service | Decouples infrastructure lifecycle from user service lifecycle; `core:` configures, never sets versions |
| `mirror:` mechanics | `mirror: <url>` → daemon creates a local bare mirror at `/opt/ownbase/repos/mirrors-<basename>` and builds from it | Declarative external mirrors, backed up locally like every other repo |
| Update model | User edits `ref:` and commits; a blank `ref:` auto-pins to the source's HEAD SHA on the next reconcile. There is no agent-opened PR flow — that mechanism was tried and removed because it added process without adding safety | Direct `ref:` commits fit the existing push→reconcile path |
| `mode:` field | Parsed but has no effect (deprecated no-op); a warning is emitted when present | Removing it outright would break existing files that reject unknown fields |
| Service image build | `localhost/ownbase-<name>:local`, built by the daemon from the repo at `ref:` | No external registry references; deterministic per ref |
| Compiler output | Deterministic, byte-identical for the same input | Golden-testable, diff-readable, no compiler state |
| Health probe | `health_probe.http: /path` → `# HealthProbeHTTP=` provenance comment in the Quadlet unit | One contract reused by dependency-gating and verified restore |
| Data volume | Named Podman volume per service (`ownbase-<name>-data`), or `volumes:` for multi-volume services with per-volume `backup:` scoping | Isolated, persistent; `backup: ["."]` is what opts a volume into the restic snapshot — omitting it excludes the volume |
| Multi-domain services | `ServiceDecl.Domains []string` alongside the older `Domain string`; `EffectiveDomains()` returns their deduplicated union; the compiler emits one Caddy route per effective domain, all pointing at the same container:port | Lets one service serve multiple public hostnames (e.g. a `.com` and a `.org`) without needing multiple service declarations; `Domain` keeps working indefinitely, no forced migration |
| Caddyfile change detection | `reconcile.DiffOptions.CaddyfileSnapshotAvailable` distinguishes "no prior snapshot" (force a reload) from "a snapshot exists and is identical" (skip) | An empty `CurrentCaddyfile` was previously ambiguous between the two, which combined with reading the snapshot *after* `compiler.WriteOutput` had already overwritten it, meant a fresh Base's Caddyfile was silently never pushed to the running Caddy container |

## SSH tunnel bridge (`ownbasectl tunnel`)

| Decision | Choice | Why |
|---|---|---|
| Agent-safe / human-run split | `create`/`vm start\|stop\|restart` never prompt for anything, ever, under any circumstance; the only interactive command anywhere in `ownbasectl` is `tunnel <name>`, which may prompt for `sudo` exactly once (mkcert's CA install) | `create`/`vm` are expected to be run by an AI agent unattended — an agent cannot answer a TTY password prompt. Starting `tunnel` is itself a human's explicit "I am sitting here, ready" signal, so an interactive prompt there is acceptable |
| No-domain Base exposure | A Base with no domain'd service exposes only SSH externally (Caddy publishes no ports, UFW opens no web ports) — this retires the old assumption that a no-domain Base gets direct, unproxied port access | There is nothing for Caddy to route to yet, and the tunnel bridge reaches services directly over SSH, so publishing ports pre-emptively only widens the attack surface for no benefit |
| Local hostname scheme | Each bridged service's real, already-configured domain with `.localhost` appended verbatim, e.g. `domain: myapp.example.com` → `https://myapp.example.com.localhost:8443` | Per RFC 6761 every OS/browser resolves any hostname *ending in* `.localhost` straight to loopback — no DNS, no `/etc/hosts`, works offline, and never changes across a VM restart/IP change, unlike `sslip.io`-style or mDNS naming (both considered and rejected — see the dev-bridge plan for the full list of alternatives and why each was rejected) |
| Certificate scope | One mkcert certificate covering the full, explicit list of `<domain>.localhost` SANs (not a wildcard) | Arbitrary production domains aren't a single-level suffix of anything shared, so `*.<name>.localhost` can't cover them; mkcert supports multiple explicit SANs in one invocation instead |
| Transport | Direct SSH tunnels to each bridged service's loopback port (`internal/tunnel`, reused unmodified), bypassing Caddy entirely | No port is firewalled on the Base — the loopback publish is only reachable via the SSH tunnel itself; works identically for a local VM or a remote server since both use the same `ServerProfile` |
| Bridge scope | Services with both a `port:` and at least one domain configured are bridged. Services marked `internal: true` are also bridged — the tunnel is their *only* access path (no Caddy route is ever emitted). A domain-less service is never bridged, and never gets a loopback publish at all | Domain-based bridging covers both publicly-reachable and intentionally-private services; `internal: true` makes tunnel-only access a first-class configuration option rather than a workaround |
| Tunnel-only (`internal: true`) services | A service with `internal: true`, `domain:`, and `port:` gets a loopback publish and is reachable via `ownbasectl tunnel`, but the compiler emits no Caddy route for it | Enables private admin UIs, dashboards, and internal APIs to be accessed securely over an authenticated SSH tunnel without ever being internet-facing — the operator explicitly opts in rather than accidentally exposing it |
| Tunnel port allocation | The compiler publishes each bridged service's container port to its own loopback port, `schema.TunnelBasePort` (41000) + sorted index among eligible services (`OwnbaseConfig.TunnelPorts()`) — never the same number as the container port | Decoupling the host-side loopback port from the container's own `port:` means a service can declare `port: 80`/`443` (colliding with Caddy's machine-wide bind) or share a port number with another service, with no collision on either side; both the compiler (rendering the Quadlet unit) and `ownbasectl tunnel` (parsing `ownbase.yaml` independently, no daemon call) compute this from the same config and always agree without coordinating |
| Local bind port | `127.0.0.1:8443` by default (`--port` to override), never `443` | Avoids a second, independent privileged-port permission requirement on top of the one-time mkcert install |
| Code iteration loop | No code-sync mechanism of any kind — no bind mount, file watcher, or hot-reload. Iterate by pushing a branch to the service's bare repo and running `ownbasectl service update --ref <branch>` (or a raw `git push` to `/opt/ownbase/repos/<name>`), exactly as in production | A second, untracked way to mutate a running Base would break the "the only mutation path is a git push" invariant (`docs/foundation/reconstruction-model.md`); this loop was already built and verified end-to-end during the Forgejo removal work |

## Secrets

| Decision | Choice | Why |
|---|---|---|
| Encryption | **age-direct** (`filippo.io/age`), not sops | One recipient per Base is sufficient for V1; sops's multi-KMS/editor-integration advantages don't apply and its dependency footprint is much larger. See [ownbase-yaml.md](ownbase-yaml.md), "Secrets". |
| Storage path | `/opt/ownbase/secrets/<service>.yaml.age`, outside git | No chicken-and-egg declaration in `ownbase.yaml`; path is deterministic from the service name |
| Injection | Start-time, scoped per service, never written to disk as plaintext | Secrets never appear in compiler output |
| Backup | `/opt/ownbase/secrets/` and `/opt/ownbase/age/` are both in every restic snapshot | Secrets aren't in git, so restic is their only persistence; key and ciphertext must travel together |

## Host hardening

| Decision | Choice | Why |
|---|---|---|
| Binary signing | **minisign** | Single binary, no PKI |
| Firewall | **UFW**, deny-all + allow SSH always, 80/443 only when `install.PassZeroConfig.ExposeWebPorts` is true (at least one service has a domain configured) | Simple, well-supported on Ubuntu; a domain-less Base has nothing for Caddy to route, so opening 80/443 would only widen the attack surface for no benefit — see "Local development" below |
| Auto-updates | `unattended-upgrades` (security-only, daily) | Standard, operator-invisible |
| Intrusion protection | `fail2ban` with an SSH jail | Well-understood, minimal config |
| Minimum OS | Ubuntu 22.04 LTS (24.04 preferred) | LTS only; no provider-specific assumptions |
| Resumability | Every hardening step is idempotent (checks before acting) | Kill mid-install, restart, continue from where it stopped |

## Backups

| Decision | Choice | Why |
|---|---|---|
| Backup tool | **restic** | Encrypts-first, deduplicated, any backend |
| Postgres PITR | pgBackRest (conditional; no-op when not installed), writing to a Podman volume that restic then backs up — pgBackRest is never configured with a remote (S3/etc.) repo directly | Single remote-store config point (`core.backup:`); no separate pgBackRest credentials to manage |
| Cadence | Backup 1h, verify 24h, retention 30 days by default | Configurable via `core.backup.interval`/`verify_interval` |
| Config vs. credentials | Repo URL in `core.backup:` (git-tracked); credentials in `/opt/ownbase/secrets/backup.yaml.age` | Config survives rebuild; credentials use the normal secrets model |
| Restorable flag | Only set **after** a verified restore drill passes | A backup never verified is not restorable by definition |
| Integrity checks | `restic check --read-data-subset=5%` + file-presence + `pg_controldata` | Practical middle ground between "ran" and "fully verified" |

## Updates and drift

| Decision | Choice | Why |
|---|---|---|
| Source-ref detection | Local git CLI against the service's bare repo (`git rev-parse`, `git rev-list --count`, `git tag --sort=-v:refname`) — never reaches upstream | The local bare repo is always current for anything already fetched; only a genuinely new ref triggers a fetch (see `internal/repos`) |
| Commit-SHA refs | Skipped in drift comparison | A 40-char hex ref is already maximally pinned |
| Blank-ref resolution | Autonomous-tier action (`update.pin_ref`) | Safe to do without asking — it only records what the source repo already is; it does not change what's running |

## Explain (status)

| Decision | Choice | Why |
|---|---|---|
| Status API | Served by the daemon at `--status-addr` (default `127.0.0.1:7070`), bearer-token auth via `--api-token` | Daemon already runs as one process; loopback-only by default |
| Operating guide | A static README.md seeded into the config repo at bootstrap (replaces the auto-init stub; never overwrites user edits). There is no generated `OWNBASE.md` — that mechanism was tried and removed | The repo holds *intent*; observed state (running/healthy, CVE counts) belongs to the status API, not to git — committing it would bury the user's history in noise commits |
| Status contract version | `schema_version` field (currently v3) guards against breaking changes | Consumers check the version before parsing |

## Integration contract

| Decision | Choice | Why |
|---|---|---|
| Build interface | `Dockerfile` at the repo root (or `dockerfile:`/`context:` in `ownbase.yaml`) | No separate manifest needed |
| Routing | All public traffic routes to the declared `port:` | Simplest model |
| Capability network | The service's key in `ownbase.yaml` is also the capability name and the network name (`ownbase-<name>-net`) | Consumers depend on the capability, not the implementation |

## Action taxonomy and audit

| Decision | Choice | Why |
|---|---|---|
| Action model | Closed enum (`ActionType` in `internal/schema/taxonomy.go`), enforced by `NewAction` | An action type that isn't in the taxonomy cannot execute; the audit log is exhaustive |
| Default tiers | Every action currently classified as `autonomous` in effect (there is no external approval device yet) | Tightening the policy later is additive, not a rewrite — see [foundation/architecture-principles.md](foundation/architecture-principles.md), principle 14 |
| Audit log format | Newline-delimited JSON, `0600` | Appendable, grep-able, not group-readable |

## Remote access (`ownbasectl`)

| Decision | Choice | Why |
|---|---|---|
| Client config | `~/.ownbase/config` (plain YAML, `0600`) | Mirrors the `~/.ssh` convention; contains tokens |
| Transport | `golang.org/x/crypto/ssh` (no `ssh` binary exec) | Pure-Go, no PATH dependency, identical behavior across platforms |
| Host key verification | `~/.ownbase/known_hosts`, trust-on-first-use | Same UX as the `ssh` CLI |
| Token bootstrap | `install.sh` generates a token, writes it to `/opt/ownbase/api-token` (root, `0600`), prints it once; `ownbasectl` fetches it over SSH automatically when adopting an existing Base | SSH is the root credential; the API token only scopes access within the tunnel it opens |
| Token hot-swap | `POST /token/reset` updates the daemon's in-memory token immediately | No daemon restart needed |
| Config mutation | `ownbasectl config get/set` and `ownbasectl service add/remove/update` all funnel through one `GET`/`POST /config` endpoint — the daemon front door for the whole `ownbase.yaml` document, validated and committed via `update.CommitFile` exactly like a manual `git push` | Single front door means every mutation path (CLI, agent, future UI) is exercised the same way the reconcile loop already trusts; no partial-update API to keep in sync with git-push semantics |
| Agent-first CLI | Every `config`/`service` subcommand is a single non-interactive invocation — no editor, no prompts, `--json` for structured output, non-zero exit on validation or transport failure | An AI agent must be able to drive the whole config lifecycle unattended |
| Secrets over the wire | Client sends plaintext over the SSH tunnel only; the daemon re-encrypts and writes to disk; no git commit | The age private key never leaves the Base; plaintext only exists in-memory for the duration of the request |
| CLI distribution | `ownbasectl` is a standalone binary (GoReleaser: darwin/linux × amd64/arm64), with `install.sh` embedded via `go:embed` so a released binary needs no checkout | `ownbasectl create` is the only supported install path — there is no `curl \| bash` |
| Daemon version pinning | Release builds of `ownbasectl` set `OWNBASE_VERSION` to their own version so `install.sh` downloads the matching signed daemon | `ownbasectl` and the daemon stay in lockstep for released installs |

## Vulnerability scanning

| Decision | Choice | Why |
|---|---|---|
| Scanner | **trivy** | Single binary, no daemon, scans both OS packages and container images, structured JSON, degrades gracefully when absent |
| Scan scope | Host OS packages + every running container image (discovered via `podman ps`) | Unbuilt images simply don't appear yet — no false "scan failed" noise |
| Cadence | 24h ticker, 5-minute initial delay after daemon start | Long enough not to hammer the trivy DB; delay lets the first reconcile finish building images |
| Remediation | `ownbasectl security fix` → `apt-get upgrade` on the Base (host packages); `ownbasectl upgrade --apply` → pull + restart (the Caddy image). Both trigger an automatic rescan | Image CVEs can only be fixed by a newer upstream image; there is no way to patch an image in place |

## Security monitoring (exposure + access)

| Decision | Choice | Why |
|---|---|---|
| Exposure scan | Local-only: `ss -tlnpH` + `ufw status` + the expected-allowlist derived from `ownbase.yaml` (SSH port, 80, 443) | Deterministic, no cloud dependency, no packets sent; answers "what does this machine believe is reachable" — an off-box scan is a possible future addition, not a requirement |
| Access monitor | `fail2ban-client status sshd` + `journalctl -u ssh` (24h window) | fail2ban and journald are already present; no new daemon, no external blocklist |
| Exposure scan cadence | At most once every 5 minutes | `ss`/`ufw`/`fail2ban-client`/`journalctl` are shell-outs; no need to re-run on every push-triggered reconcile |
| Graceful degradation | Missing tools (`ss`, `ufw`, etc.) return `Available=false` + zero values, never an error | Keeps status queries working on machines where a probe tool is temporarily unavailable |
